# Final Manuscript Model-Code Alignment Design

## Goal

Rebuild the research code so the executable implementation matches the final manuscript's mathematical model, algorithm design and pseudocode. Legacy result files remain available but are not treated as final manuscript evidence.

## Scope

The implementation will follow the final manuscript, not the old order-only scheduler. The code must include:

- bounded same-stop sortie pattern set \(\mathcal K_s\);
- q95 feasible service set \(\mathcal F\) and service graph \(\mathcal G_K\);
- explicit schedule state with vehicle route, stop-batch sortie lists, UAV assignment, launch time, recovery time, vehicle arrival/departure, tower start/completion and missed-service diagnostics;
- objective evaluator for \(F(x)\) with RWCT, makespan, ground travel, q95 energy and missed-service penalty;
- RV-ALNS that operates on the schedule state, not only a tower order;
- destroy, repair, pattern split/reselection, local search, simulated-annealing acceptance and adaptive operator weights;
- refreshed experiment outputs generated by the new implementation.

## Code Boundaries

New model code will be placed under `src/uqrv/`:

- `stop_batch_model.py`: manuscript-level data structures, service graph construction, schedule evaluation and objective breakdown.
- `rv_alns.py`: schedule-state ALNS operators and solver.

Existing files will be adapted instead of removed:

- `src/uqrv/solvers.py`: route manuscript-facing ALNS methods to the new solver while keeping simple baselines.
- `scripts/run_publishable_experiments.py`: mark new runs with implementation metadata and use the rebuilt solver paths.
- `scripts/write_full_result_tables.py`: continue consuming result CSVs after rerun.

## Documentation Scope

The documentation will be written in a software-reference style but will not include installation, development workflow or architecture chapters. Required documentation pages:

- mathematical model;
- algorithm and pseudocode mapping;
- data structures and API;
- experiment reproduction and result evidence;
- claim-to-code evidence ledger.

## Evidence Rule

Final results are valid only if generated after this rebuild. The evidence ledger must map each retained manuscript claim to code path, command and result file. If a claim has no evidence, it must not be treated as supported.

## Validation

Validation must include:

- unit tests for service graph construction, q95 filtering, schedule timing, objective terms and ALNS diagnostics;
- full `RUN_TESTS.ps1`;
- at least one main experiment rerun before reporting new manuscript numbers;
- claim/evidence ledger updated with new result paths.
